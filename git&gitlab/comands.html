Comandos de Git y Gitlab
git config » Muestra las opciones de configuracion.
Banderas:
--list » Muestra la configuracion por defecto de git.
--list --show-origin» Muestra la ubicación de los archivos de configuración de git avanzada.
--global user.name "el nombre" » Nos permite modificar el nombre de usuario que posee Git en nuestra configuracion global.
--global user.email "el correo" » Nos permite modificar el nombre de correo electronico que posee Git en nuestra configuracion global.
git init » Inicia el seguimiento de los cambios en una rama de archivos.
git status » Muestra el estatus de los archivos nuevos o modificados en memoria de "Staging".
git add » Agrega los archivos nuevos o modificados a la memoria de "Staging" (memoria chache), algunas de sus banderas pueden ser: "archivo.txt", ., -A, -am (Agregar todos).
git commit » Agrega los archivos a la memoria del disco duro, asignandole un identificador al/los archivo(s), usa como parametro -m + "descripcion" para asignar una descripcion a los cambios realizados, de lo contrario se ejecutara el editor de texto de consola VIM. Otro parametro es -a que nos permite agregar el archivo a la memora de staging (git add) y luego editarlo en VIM. Tambien se puede hacer uso de ambas banderas -am + "descripcion".
git log » Muestra todo el historial de cambios en la rama especifica (HEAD).
--stat » ofrece informacion con mas detalles de los cambios.
--all » Muestra todos los commits.
--graph » Dibuja las fusiones (merge) que se han realizado entre ramas.
--decorate » No se.
--oneline » Lista los commits en una sola linea.
git show » Muestra el texto del archivo que ha sido modificado.
git branch » Muestra todas las ramas de trabajo y donde se encuentra ubicado.
git rm --cached + "archivo" » Elimina el archivo de la memoria de staying(RAM) para ser anadido.
git rm --force + "archivo" » Elimina el archivo de la memoria de staying(RAM) y del directorio donde fue creado antes de ser comentado.
git reset » HEAD Elimina los archivos de la memoria de staying(RAM) para luego ser anadidos. Otras banderas --soft Borra todo el historial y los registros de Git pero guardamos los cambios que tengamos en Staging, así podemos aplicar las últimas actualizaciones a un nuevo commit. --hard Borra todo. Todo todito, absolutamente todo. Toda la información de los commits y del área de staging se borra del historial.
git diff + "idcommit1(viejo)" "idcommit2(nuevo)" » Compara los cambios que han ocurrido en un archivo (tomar en cuenta el orden).
git checkout » Nos permite viajar en el tiempo utilizando como parametro los "idcommit", tambien nos permite cambiar de rama de trabajo asignando como parametro el "nombre de la rama"
git clone + "URL" » Nos permite clonar un repositorio remoto.
git push » Luego de hacer git add y git commit debemos ejecutar este comando para mandar los cambios al servidor remoto.
git fetch » Lo usamos para traer actualizaciones del servidor remoto y guardarlas en nuestro repositorio local (en caso de que hayan, por supuesto).
git merge + "nombre de la rama" » También usamos el comando git fetch con servidores remotos. Lo necesitamos para combinar los últimos cambios del servidor remoto y nuestro directorio de trabajo. Se debe utilizar estando en la rama principal (master) para ejecutarlo.
git pull » Básicamente, git fetch y git merge al mismo tiempo.
git remote add origin “URL(HTTPS)” » Nos permite asignar una url de origen (Github) del proyecto o repositorio remoto a nuestro repositorio local (Git).
git remote » Una vez asignada la URL de origen este comando nos muestra “origin”.
Banderas:
-v » Nos permite ver la URL contenida en origin.
git push origin master » Nos pide autenticacion del usuario romoto en Github, si el login es correcto nos envia nuestros cambios o modificaciones realizadas en el repositorio local al repositorio remoto a una rama especifica, en este caso: master.
git pull origin master » Nos permite actualizar o fusionar nuestro repositorio local junto con el repositorio remoto. En ocaciones debemos utilizar la bandera:
--alow-unrelated-histories » Nos permite fusionar el historial de cambios (remoto-local).
ssh-keygen -t rsa -b 4096 -C "ejemplo@email.com" » Ubicados en el directorio principal “Home / ~”, este comando nos permite crear una llave SSH.
Banderas:
-t » Nos permite seleccionar el algoritmo que deseamos utilizar.
-b » Seleccionamos la complejidad desde una perspectiva matematica (curso de algoritmos) de la llave.
-C » Nos permite asignar un correo electrónico (Debe coincidir con el que se esta usando en Github).
eval $(ssh-agent -s) » Evalua si el proceso esta en ejecucion y retorna el PID (Process ID)
ssh-add "ruta donde se guardan las claves SSH" » Agrega como identidad nuestra llave privada.
git remote set-url origin "URL-SSH" » Nos permite cambiar nuestra URL de origen del repositorio en la configuracion de git, recordando que debemos estar dentro de la carpeta del proyecto. Si los archivos de llaves publicas tienen contrasena nos pedira colocarlas.
Terminologia de Git
  • Archivos Tracked: Son los archivos que viven dentro de Git, no tienen cambios pendientes y sus últimas actualizaciones han sido guardadas en el repositorio gracias a los comandos git add y git commit.
  • Archivos Staged: Son archivos en Staging. Viven dentro de Git y hay registro de ellos porque han sido afectados por el comando git add, aunque no sus últimos cambios. Git ya sabe de la existencia de estos últimos cambios, pero todavía no han sido guardados definitivamente en el repositorio porque falta ejecutar el comando git commit.
  • Archivos Unstaged: Entiéndelos como archivos “Tracked pero Unstaged”. Son archivos que viven dentro de Git pero no han sido afectados por el comando git add ni mucho menos por git commit. Git tiene un registro de estos archivos, pero está desactualizado, sus últimas versiones solo están guardadas en el disco duro.
  • Archivos Untracked: Son archivos que NO viven dentro de Git, solo en el disco duro. Nunca han sido afectados por git add, así que Git no tiene registros de su existencia.
  • Raw: Muestra el código plano del archivo.
  • Blame: Muestra los usuarios responsables de los cambios en los archivos.
  • History: Muestra el historial de cambios del archivo (git log).
  • Passphrase: Cuando vamos a asignar una contrsena para las Claves SSH debemos agregar espacios.
  • SSH: Secure Shell.
